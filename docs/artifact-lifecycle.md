# Artifact Lifecycle

**Version:** 1.1  
**Date:** 2026-03-30  
**Status:** Supporting doc

This repository is meant to absorb and refine artifacts that have proven useful in real projects.

## Promotion Rule

Templates, examples, skills, and supporting notes should usually be promoted into this repository only after they have been used outside the repository in at least one real project and improved through that use.

## Artifact Types

- **Method**: the vendor-neutral process and its governing principles
- **Template**: a blank artifact intended for reuse
- **Example**: a filled artifact showing how the method looks in practice
- **Vendor Pack**: a vendor-specific extension layered on the method
- **Skill**: an instruction artifact intended for a specific AI role in a specific tool context

## Promotion Flow

1. Try a pattern in a real project.
2. Revise it based on friction, omissions, or ambiguity.
3. Generalize it enough to be reusable.
4. Place it in the appropriate directory in this repository.
5. Document why it exists and when it should be used.

## Quality Bar

An artifact is ready to promote when:

- it solves a recurring problem
- it can be understood without project-specific tribal knowledge
- it does not accidentally encode vendor-specific assumptions in core material
- it is short enough to be practically reusable

## Maintenance Expectations

- Templates may change as better defaults emerge.
- Examples should be added sparingly and only when they teach something new.
- Vendor packs may evolve faster than the core method.
- Historical artifacts can be kept if they explain the evolution of the method, but active guidance should remain clear and current.
